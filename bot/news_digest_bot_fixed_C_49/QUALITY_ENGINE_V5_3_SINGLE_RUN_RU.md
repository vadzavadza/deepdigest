# Quality Engine v5.3 — single-run adaptive search

Цель v5.3: исправить поведение, когда первый запуск темы даёт 0–1 новость, а второй запуск через минуту внезапно находит ещё одну. Теперь один запуск должен сам сделать несколько поисковых проходов в рамках бюджета, собрать общий пул кандидатов, отфильтровать, дедуплицировать и только потом отправлять дайджест.

## Что изменено

### 1. Больше поисковых углов за те же деньги

Было:

```env
OPENROUTER_WEB_SEARCH_RESULTS_PER_CALL=8
OPENROUTER_WEB_SEARCH_MAX_CALLS_PER_TOPIC=2
```

Стало:

```env
OPENROUTER_WEB_SEARCH_RESULTS_PER_CALL=5
OPENROUTER_WEB_SEARCH_MAX_CALLS_PER_TOPIC=3
```

При цене `$0.004` за результат это примерно:

```text
3 × 5 × 0.004 = $0.06
```

То есть остаёмся ниже потолка `$0.07`, но получаем не два, а три разных query-pass.

### 2. Single-run adaptive orchestration

OpenRouter source теперь логирует и выполняет поисковые проходы как единый сбор кандидатов:

```text
pass_1: raw / accepted / added
pass_2: raw / accepted / added
pass_3: raw / accepted / added
final merged count
```

Если после первых проходов кандидатов мало, rescue/snippet fallback включается внутри этого же запуска, а не только после повторного нажатия пользователем.

### 3. Менее “самоцензурный” prompt

Раньше prompt просил OpenRouter не возвращать ссылки без видимой даты. Это могло приводить к нулям, потому что дата часто не видна в search annotation. Теперь OpenRouter отдаёт кандидатов, а локальный слой сам проверяет дату, тип страницы, directness и freshness.

### 4. Детерминированнее web-search запрос

В payload добавлены:

```json
"temperature": 0,
"top_p": 0.2
```

Это должно снизить разброс между первым и вторым запуском.

### 5. Нишевые источники не режутся слишком рано

До v5.3 часть esports/technology/game/entertainment ссылок могла отлетать ещё на этапе извлечения из search annotations из-за нестандартного URL. Теперь ранний фильтр стал мягче, а строгая проверка перенесена в локальный quality gate.

### 6. Better query ordering

Для tech/esports/game-подобных тем второй и третий поисковые проходы теперь получают не просто `breaking news`, а семантические расширения. Например для esports-CS тем в первых проходах появляется `CS2 Counter-Strike`, а для software topics — `software release`, `security`, `developer ecosystem`.

Это не хардкод под одну тему, а общий принцип: короткие/нишевые запросы требуют контекстного расширения.

### 7. Immediate repeat guard

Если пользователь нажимает “Запустить сейчас” повторно сразу после успешного дайджеста, бот больше не должен доставать старые остатки из того же окна. Включён guard:

```env
IMMEDIATE_REPEAT_GUARD_MINUTES=15
```

В этот период повторный запуск ищет только реально более новые материалы, а не “второй сорт” из прошлого окна.

### 8. Дедуп спортивных матчей на кириллице

Расширены event terms для украинских/русских match-report заголовков. Это должно лучше склеивать материалы типа “Шахтар победил Зорю” и “Шахтар дотиснув Зорю...” в один сюжет.

## Что не менялось

- Старые новости всё ещё не должны проходить как fresh.
- Verified publish date остаётся главным сигналом.
- Search-snippet fallback остаётся ограниченным и штрафуется.
- Бюджетный потолок `$0.07` сохранён.
- Рабочие файлы не удалялись.

## Что смотреть в тестах у себя

1. Первый запуск должен чаще сразу давать полный доступный дайджест, а не требовать второго клика.
2. Повторный запуск через минуту после успешного дайджеста должен чаще отвечать “новых нет”, а не слать слабые остатки.
3. Темы вроде `python`, `BLAST CS`, `world of tanks` должны реже уходить в 0 из-за раннего URL-фильтра.
4. Старые статьи всё ещё не должны пролезать.
5. News hub pages всё ещё должны резаться.
